## 3. 根因诊断: 场景设计缺陷（非算法问题）

### 结论: **不是 MAPPO 的问题，是场景机制让进攻方物理上不可能突防。**

以下按严重程度排序列出 9 个关键问题:

---

### 🔴 致命问题 #1: `init_dist` 归一化错误

**问题**: `reward_cost.py` 中 `init_dist = 5000.0`，但 V23 实际进攻→HVT 距离 ≈ **2400m**

**影响**: 所有距离归一化的奖励信号衰减约 2 倍
- `approach_hvt_reward`: $r = 2500 \times \frac{\Delta d}{5000}$ → 应该是 $\frac{\Delta d}{2400}$
- `proximity_reward`: $(1 - \frac{d}{5000})^3$ → 起始值 $(0.52)^3 = 0.14$，几乎为零
- `progress_reward`: $1 - \frac{d}{5000}$ → 起始值仅 0.52

**修复**: `init_dist = 2400` 或动态计算为 `||off_center - hvt||`

---

### 🔴 致命问题 #2: `action_scale = 0.5` 严重限制机动能力

**问题**: 进攻方 `ny_max = ±5g`，但 `action_scale = 0.5` 将其截断为 **有效 ny = ±2.5g**

**对比**:
| | 进攻方 (实际) | 拦截器 |
|---|---|---|
| 横向过载 | ±2.5g | ±8g |
| 最小转弯半径 | 102m | 62m |
| 机动比 | 1× | **3.2×** |

拦截器的机动性是进攻方的 **3.2 倍**！无论什么算法，进攻方物理上无法通过机动闪避拦截。

**修复**: `action_scale = 1.0` 或更大

---

### 🔴 致命问题 #3: 拦截器永不放弃 (V23 变更)

**问题**: V23 移除了 `MISSED`/`ABANDONED` 状态，拦截器在脱离后立即重新锁定最近进攻方

**影响**: FOV 逃逸毫无意义 — 逃逸后拦截器立即再次追击。这等于把 fov_escape_bonus = 200 的奖励变成了"空头支票"。

**修复**: 恢复 miss_cooldown（例如 500 步 / 5 秒冷却），让逃逸有真实战术价值

---

### 🟠 严重问题 #4: CPA 互杀 + 高闭合速度 = 拦截几乎必然成功

**闭合速度**: 50 + 70 = **120 m/s** (最差情况迎头)
**CPA 杀伤判定**: 沿相对速度方向采样 10 点，< 5m 即互杀
**dt = 0.01s**: 每步位移 0.5m，10 点覆盖 5m → 有效扫过 10m 圆柱

在 120m/s 闭合速度下，进攻方需要在 ~0.04s 内完成 5m 的侧向位移才能躲避。
以 2.5g (24.5 m/s²) 侧向加速度，0.04s 内侧移仅 $\frac{1}{2} \times 24.5 \times 0.04^2 = 0.02m$ — **远不够**。

**修复建议**:
- 增大 `hit_hvt_range` 到 15-20m（为进攻方降低精度要求）
- 或减小 `collision_kill_range` 到 2m
- 或增加拦截器的 miss 惩罚/冷却

---

### 🟠 严重问题 #5: `engagement_range = 20m` 窗口太窄

在 120m/s 闭合速度下：
- 从 20m 到 0m 仅需 **0.17 秒 (17 步)**
- 进攻方需要在 17 步内完成 FOV 逃逸 → 几乎不可能

**修复**: `engagement_range = 150-200m`

---

### 🟡 中等问题 #6: 速度 50 vs 70 m/s (1.4× 劣势)

拦截器速度 = 进攻方的 **1.4 倍**。进攻方无法甩开拦截器。

**修复**: 将进攻方速度提升到 60-65 m/s，或降低拦截器到 60 m/s

---

### 🟡 中等问题 #7: 高度惩罚矛盾

`z_min = 0`，`z_min_safe = z_min × 2.0 = 0`
- 低空惩罚实际上**失效了**（z_safe = 0 时分母为 0）
- 但 HVT 在 z = 0，进攻方需要俯冲才能接近
- 同时有 `dive_penalty`：γ < -5° 时受惩罚

进攻方被鼓励保持高度 (nz 均值 = 0.40 = 持续爬升)，但目标在地面。

**修复**: 在接近 HVT 时减弱/取消俯冲惩罚

---

### 🟡 中等问题 #8: 里程碑距离未校准

| 里程碑 | 需要前进(m) | 耗时(理论/s) | 已到达? |
|--------|-----------|-------------|---------|
| 1500m | 900 | 18s | ✅ 部分agent |
| 1000m | 1400 | 28s | ❌ 几乎没有 |
| 500m | 1900 | 38s | ❌ 完全没有 |
| 200m | 2200 | 44s | ❌ 完全没有 |

从评估结果看，最近距离 811m，说明 1000m 里程碑几乎不被触发。

---

### 🟢 次要问题 #9: 奖励信号密度不足

dt = 0.01 每步位移仅 0.5m，approach reward 每步仅 $\sim 2500 \times \frac{0.5}{5000} = 0.25$

同时 `step_penalty = -0.005` × 4 agents = -0.02/步，`detected_penalty = -0.005` × detected agents

奖励稀疏性不是主要问题，但会减慢学习速度。

## 4. 为什么不是算法问题？

| 考量 | 分析 |
|------|------|
| MAPPO vs MACPO | MACPO 增加了安全约束（cost限制），但**当前问题是进攻方物理上无法突防**。安全约束不会帮助进攻方飞得更快或更灵活 |
| 奖励设计 | 进攻方的 approach reward 被 init_dist=5000 削弱了一半，且 proximity reward 在整个战场上接近 0 |
| 学到了什么 | 策略学会了：(1) 保持少量机动避免被杀 (2) 触发 FOV 逃逸获取 escape_bonus (3) 尽量生存到超时 |
| 为什么不朝HVT飞 | 因为朝 HVT 飞 = 朝拦截器飞 = 被杀。被杀惩罚 (-3.0/步) 远大于前进奖励 (~0.25/步)。策略**理性地选择了存活率更高的保守行为** |

## 5. 修复建议 (优先级排序)

### Phase 1 — 紧急修复（预计效果: 成功率 0% → 30-50%）

```python
# reward_cost.py
init_dist = 2500  # ← 从 5000 改为实际距离

# config.py
"action_scale": 1.0,              # ← 从 0.5 改为 1.0
"engagement_range": 150,           # ← 从 20 改为 150
"miss_cooldown_steps": 500,        # ← 恢复拦截器冷却 (5秒)
```

### Phase 2 — 平衡调整（预计效果: 30% → 60%+）

```python
# config.py - offensive
"v_nominal": 60.0,                 # ← 从 50 提升到 60
"v_max": 80.0,                     # ← 从 70 提升到 80

# config.py - defensive
"v_nominal": 65.0,                 # ← 从 70 降低到 65

# config.py - kill mechanics
"hit_hvt_range": 10.0,             # ← 从 5 放大到 10
"collision_kill_range": 3.0,       # ← 从 5 减小到 3
```

### Phase 3 — 奖励优化（微调）

```python
# reward_cost.py
approach_hvt_coef = 3500           # ← 增强接近奖励
proximity_reward_coef = 10.0       # ← 增强近距奖励
killed_penalty = -1.5              # ← 减轻被杀惩罚，鼓励冒险
milestone_bonuses = {1800: 50, 1200: 100, 600: 200, 200: 400}  # 重新校准
```

### Phase 4 — 算法考虑

**仅在 Phase 1-3 完成后**，如果成功率仍低且发现 cost 违规问题（如频繁飞出边界、碰撞），再考虑切换 MACPO。

---

## 6. 总结

**核心结论**: V23 突防训练失败的根本原因是 **场景物理设计让进攻方不可能成功突防**，而非算法选择问题。

进攻方面临:
- 机动性仅为拦截器的 **1/3.2** (2.5g vs 8g)
- 速度仅为拦截器的 **71%** (50 vs 70 m/s)
- 拦截器**永不放弃追击**
- FOV 逃逸窗口仅 **0.17秒**
- 奖励信号被 **init_dist bug** 衰减了 ~2倍

策略的"不前进"行为是**理性最优**：前进 = 被杀，保持距离 = 存活获得更多 step reward。

修复场景设计后重新训练，预计 MAPPO 即可实现合理的突防成功率。
